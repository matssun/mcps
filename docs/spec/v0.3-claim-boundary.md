# MCP-S v0.3 — Tiered Multi-Node Profile Within One Trust Domain

*The v0.3 epic / claim-boundary document. Composes ADR-MCPS-020 through ADR-MCPS-023 and sets the boundary of what the v0.3 multi-node release may claim.*

## Summary

MCP-S v0.3 extends the single-node Rust-native profile to **multi-node deployments within one trust domain / one operator**.

This epic is the umbrella **claim-boundary document** that composes ADR-MCPS-020 through ADR-MCPS-023 into one coherent release story. It is not only a scheduling container.

> The v0.3 epic does not create one unconditional “multi-node secure” claim. It creates a **tiered multi-node profile** whose security claim is determined by the selected replay-store durability tier, trust-propagation tier, key-custody mode, and ingress mode.

A tier appearing in an ADR does not by itself mean that tier is implemented or claimable in v0.3. The v0.3 release claim may use only tiers that are implemented, configured, documented, and covered by conformance tests.

## The v0.3 claim depends on four declared deployment modes

| # | Declared mode                       | ADR          | Determines                                                                                                  |
| - | ----------------------------------- | ------------ | ----------------------------------------------------------------------------------------------------------- |
| 1 | Replay-store durability tier        | [ADR-MCPS-020](../adr/adr-mcps-020.md) | `REDIS_ASYNC` / `REDIS_WAIT_QUORUM` / `LINEARIZABLE` / `SINGLE_STORE_FAIL_CLOSED` — the replay-safety claim |
| 2 | Trust-propagation window `T` / tier | [ADR-MCPS-021](../adr/adr-mcps-021.md) | revocation enforcement window; bounded-cache / live / push                                                  |
| 3 | Signing key-custody mode            | [ADR-MCPS-022](../adr/adr-mcps-022.md) | `per_node_keyset` / `shared_remote_signer`; custody versus blast radius                                     |
| 4 | Ingress / transport-binding mode    | [ADR-MCPS-023](../adr/adr-mcps-023.md) | `end_to_end_mtls` / `trusted_ingress_asserted`                                                              |

The claim changes with the selected tier. Redis async failover, Redis `WAIT`, and a CP/linearizable store do not support the same replay claim. Trusted-ingress mode is not end-to-end channel binding. Shared KMS identity does not have the same blast-radius profile as per-node keys.

* [ ] tests proving each implemented and supported tier behaves as claimed

## Not in v0.3

The following would be over-claims unless separately scoped, implemented, and tested:

* multi-tenant isolation between mutually distrusting operators or tenants;
* unconditional replay safety on async Redis failover;
* zero-window revocation unless a live/linearizable or reliable monotonic push tier is configured;
* end-to-end channel binding when running in `trusted_ingress_asserted` mode;
* smaller blast radius for shared KMS identity than for per-node keys;
* copying one private key across nodes, which is forbidden;
* Tier-3 LB-signed request-bound ingress assertions;
* hostile shared-store threat model;
* cross-operator replay-store isolation;
* public/upstream MCP extension status.

## Release-gate rule

The v0.3 multi-node claim turns on only when all of the following are true:

1. ADR-MCPS-020 through ADR-MCPS-023 are accepted.
2. The selected v0.3 supported tiers are implemented.
3. `security-boundary.md` composes the four ADRs into one explicit v0.3 claim.
4. The conformance manifest lists the implemented tiers and their tests.
5. CI is green for the v0.3 conformance suite.
6. The documentation states the exact allowed claims and forbidden claims for each supported tier combination.

ADR acceptance alone does not enable the v0.3 claim.

## Process

Design sketches are reviewed in the ADR PR. Per the project’s design-PR-then-implementation-PR rhythm, each ADR’s implementation lands as its own follow-up PR once the design is accepted.

The `security-boundary.md` composition is the release gate. The v0.3 multi-node claim turns on only when the four ADRs, their implementations, the composed boundary, and the conformance vectors for the implemented tiers are merged and green.

## Default posture to be confirmed

The implementation PRs must declare the default v0.3 baseline before the release claim is enabled.

Candidate baseline:

* replay: `REDIS_WAIT_QUORUM` where configured, otherwise `REDIS_ASYNC` with bounded failover caveat;
* trust propagation: bounded `T`, default 60 seconds;
* key custody: `per_node_keyset`;
* ingress: `end_to_end_mtls`.

If a deployment selects a weaker or different tier, the security claim must be downgraded accordingly.

## Alignment with the MCP 2026-07-28 release (RC)

The upstream MCP 2026-07-28 release candidate moves the protocol to a stateless,
horizontally-scaled model. This is the same deployment shape v0.3 targets, so it
mostly confirms the v0.3 direction; three composition deltas are captured as
**conditional** Delta ADRs (they revisit if the referenced SEP schema changes
materially before the spec is Final):

| Delta ADR | Driver | What it pins |
| --------- | ------ | ------------ |
| [ADR-MCPS-024](../adr/adr-mcps-024.md) | SEP-2322 multi round-trip | each leg is an independent signed request; `requestState` is never trusted unsigned; fresh nonce per leg/retry |
| [ADR-MCPS-025](../adr/adr-mcps-025.md) | SEP-2243 routing headers | `Mcp-Method`/`Mcp-Name`/`x-mcp-header` are untrusted routing hints; the signed body decides; fail closed on header↔body disagreement |
| [ADR-MCPS-026](../adr/adr-mcps-026.md) | SEP-2575 stateless `_meta` | decision-influencing per-request `_meta` (protocol version at minimum) is in signing scope; observability `_meta` is not |

Confirmations (no delta needed):

- **SEP-2567 (sessions removed):** v0.3 reintroduces no session affinity — MCP-S
  signs per request, so the stateless model is the assumed baseline.
- **SEP-2484 (conformance gating):** the ADR-MCPS-011 conformance vectors are
  structured to map onto upstream conformance scenarios if MCP-S enters the
  SEP-2133 Extensions Track.
