## B.1 — JSON number domain (integer-only vs full RFC 8785 floats)
**Claude rec:** keep integer-only for v1; document limitation; floats → future scheme.
**Codex answer (summary):** EXPAND to full RFC 8785 floats now. "MCP payloads naturally contain temperatures, coordinates, prices, scores, sampling parameters… A 'secure' envelope that cannot protect normal MCP payloads… is not conservative security; it is deferred failure." `…-json-v1` should accept the full RFC 8785 number domain with no fallback formatter; allowlist handles future migration. Required exhaustive IEEE-754 vectors (`-0`, `1e21`, subnormals, shortest-round-trip ties, equivalent-lexeme `1`=`1.0`=`1e0`, signed-request float vectors, tamper vectors, NaN/Inf rejection, cross-impl oracle corpus asserting bytes+hashes).
**Mats (override → resolved `[user, Codex-overridden]`):** KEEP integer-only for v0.6; make it an explicit, named, tested limitation; do not imply full RFC 8785. Rename scheme `mcps-jcs-int53-json-v1`. Document the hole honestly (proposal/ADR/conformance/release notes) and test the failure.

### Staged ADR text from Mats (insert only on summary approval)
> ### B.1 Number domain decision — integer-only for v0.6
> MCP-S draft-02 / v0.6 keeps the draft-01 integer-only JSON number domain for the first canonicalization scheme. The scheme accepts only JSON integers within ±(2^53 − 1). Fractional numbers, exponent-form numbers, NaN/Inf, and numeric coercions are rejected before signature verification with canonicalization failure. This is an intentional scope boundary, not a parser limitation.
> Rationale: full RFC 8785 fractional-number serialization (ECMAScript number-to-string) is one of the highest-risk areas for independent implementations to disagree byte-for-byte; the first security-critical scheme chooses the stricter domain.
> Consequence: MCP-S v0.6 does not protect messages whose signed payload contains JSON fractional numbers (e.g. `{"temperature":0.7}`, `{"price":19.99}`, `{"latitude":55.7047}`); they fail closed unless represented outside the JSON number domain or handled by a later scheme.
> Scheme id must make the restriction visible — candidate `mcps-jcs-int53-json-v1`. A later scheme may support full RFC 8785 numbers under a separate id with exhaustive vectors before any verifier profile allows it.
> Required vectors: integer accepted; max-safe-int accepted; min-safe-int accepted; above-max rejected; below-min rejected; fractional rejected; exponent rejected; `-0` rejected/normalized per exact rule; `19.99` rejected; `0.7` rejected.

## C — observability / preimage exclusion set
**Claude rec:** keep the asymmetry; exclusion set unchanged from draft-01; canon_id/version protected.
**Codex answer (summary):** KEEP the container-vs-nested asymmetry — it's the stricter, correct boundary. Formalize as explicit JSON-path predicate: EXCLUDE only (a) envelope `signature/value` (req `/params/_meta/se.syncom~1mcps.request/signature/value`, resp `/result/_meta/.../signature/value`) and (b) the three W3C keys `traceparent`/`tracestate`/`baggage` ONLY at container-level `params._meta`(req)/`result._meta`(resp). Nothing recursive, nothing by key-name alone, nothing in arguments/content/nested `_meta`/envelope. No other exclusions; `version`/`canonicalization_id`/alg/key_id/authorization_hash/request_hash/claims/nonce/timestamps/method/params/arguments/result all signed; unknown fields REJECTED (deny_unknown_fields), not excluded. 9 vectors incl. container-rewrite-still-verifies, nested-rewrite-fails, both-present split, mutate version/canon_id/alg/key_id fails, independent byte-equality vector (assert bytes not print, S8).

Keep-vs-redefine has a defensible engineering answer (not Trigger 4); strict container-only predicate aligns S14/S2/S5; vectors satisfy S8 anti-gaming (byte-equality, both directions, independent recompute); deny_unknown_fields fail-closed; no new vocabulary, in scope. → `[Codex, judge-passed]`.

## E.2 — authz-system-reference field schema (Q10/Q11/Q13)
**Claude rec:** require a self-contained digest (S16) + decision id as cross-reference; escalate field names.
**Codex answer (summary):** REQUIRE BOTH — all six mandatory, none optional: `{binding_type:"authz-system-reference", authorization_system_id, reference_scheme_id, reference_value, digest_alg:"sha256", digest_value}` (b64url-no-pad, NO `sha256:` prefix since alg split into digest_alg). S16 rationale: digest-alone = no audit route to the decision; reference-alone = live-system dependency → if authz system purges/rotates the record the signed MCP-S evidence becomes non-reconstructable = DEFECT under S16. So digest mandatory + self-contained cryptographic commitment (historical verification from signed record + archived external evidence + trust material valid at the time, independent of external live DB); reference_value mandatory as cross-reference, not the binding. Min auditable metadata = exactly those fields; don't duplicate envelope signer/key_id/issue-time/version/canon_id. §5 preserved: authz system computes digest under reference_scheme_id; MCP-S binds presence, never recomputes/interprets.
**Judge:** ESCALATE, Trigger 4 only. Substance auto-passes — "require both, all mandatory" = strict no-fallback S4/S14; S16 durability rationale matches D1; digest split consistent with opaque form. Escalate only the NEW protected field names (S10 — Mats's call).
